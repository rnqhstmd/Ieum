## 1. 전략 선언 — 기능 수 경쟁 금지

> **Ieum은 노션과 기능 개수로 경쟁하지 않는다.**
> **자체 CRDT 아키텍처에서 거의 공짜로 나오는, 노션이 구조적으로 약한 기능을 무기로 삼아
> "실시간·로컬 우선 협업 에디터"로 포지셔닝한다.**

노션은 수백 명의 팀이 수년에 걸쳐 쌓은 기능을 보유한다. 블록 타입, 데이터베이스, 워크플로우 자동화 — Ieum이 이 축에서 경쟁하면 출발 전에 진다.

반면 노션의 협업 엔진은 OT 기반이거나 서버 중심 설계로 알려져 있으며, 구조적으로 오프라인이 취약하고 버전 히스토리가 비싸다. Ieum의 자체 RGA CRDT는 이 세 가지를 **아키텍처 수준에서** 해결한다 — 추가 개발 비용이 거의 없다. 이것이 wedge(쐐기)다.

**원칙**
- 기능 카탈로그가 아닌 **핵심 경험 3개**로 포지션을 세운다.
- 아키텍처 우위가 UX로 체감되는 기능을 먼저 출시한다.
- 노션이 구조적으로 따라잡기 어려운 기능에 마케팅 자원을 집중한다.

## 2. 차별점 3계층

### 계층 분류 기준

| 계층 | 설명 | 전략적 가치 |
|------|------|------------|
| **[A] 아키텍처에서 공짜로 나오는 것** | CRDT op 로그·RGA id 앵커링·오프라인 병합이 이미 구조에 내재 | 최강. 노션이 재아키텍처 없이는 따라잡기 어렵다 |
| **[B] 협업 UX (피그마급 멀티플레이어)** | P2~P3 인프라 위에 구현. 중간 난이도 | 강함. 차별화 가시성이 높고 바이럴 각도가 있다 |
| **[C] 더 무거운 것 (후순위)** | 독자 구현 복잡도 높고 차별성 약함 | 약함. AI 요약 등은 이미 업계 표준 |

---

### [A] 아키텍처에서 공짜로 나오는 것

#### A-1. 로컬 우선 · 오프라인 편집 ⭐ wedge

| 항목 | 내용 |
|------|------|
| **무엇인가** | 인터넷 연결이 없어도 편집이 가능하고, 재연결 시 충돌 없이 자동 병합된다. 입력이 즉시 화면에 반영된다(optimistic local-first). |
| **노션의 약점** | 노션은 오프라인 편집을 공식 지원하지 않거나 매우 제한적이다. 연결이 끊기면 편집이 막히거나 나중에 덮어쓰기 충돌이 발생한다. 입력 응답성도 서버 왕복에 의존한다. |
| **CRDT가 가능케 하는 이유** | RGA는 네트워크 없이도 로컬에서 op를 생성·적용할 수 있다. 재접속 시 append-only op 로그를 교환하면 알고리즘이 자동으로 수렴시킨다. 서버는 단순 relay일 뿐이다. |
| **구현 난이도** | ★★★☆☆ — CRDT 코어(P2)는 오프라인 병합을 이미 지원한다. 추가 작업은 op 큐의 로컬 영속화(IndexedDB)와 재연결 재전송 로직이다. |
| **적합 Phase** | P2-P3 연장(코어 오프라인 병합) / 완전한 local-first 영속화는 P5+ |
| **데모·바이럴 각도** | "비행기 모드에서 노트 작성 → 착륙 후 팀원 편집과 충돌 없이 병합" 시연. 노션과 나란히 비교 영상. |

---

#### A-2. 무한 버전 히스토리 + 타임머신 ⭐ wedge

| 항목 | 내용 |
|------|------|
| **무엇인가** | 문서의 모든 편집 이력을 글자 단위로 보존하고, 임의 시점으로 되감기·재생이 가능하다. |
| **노션의 약점** | 노션 버전 기록은 유료 플랜에서만 사용 가능하고, 블록 단위 diff를 보여주지 않으며, 임의 시점 복원이 거칠다. |
| **CRDT가 가능케 하는 이유** | CrdtOp는 append-only 로그로 저장된다(P2부터). 특정 시점의 op 집합만 재생하면 그 순간의 문서 상태가 복원된다. 추가 저장 비용 없이 "데이터는 공짜"다. |
| **구현 난이도** | ★★☆☆☆(데이터) / ★★★☆☆(UI) — op 로그는 P2에서 이미 쌓인다. 히스토리 탐색 UI와 재생 로직만 추가하면 된다. |
| **적합 Phase** | P5(post-MVP) — 데이터는 P2부터 쌓이므로 "데이터는 공짜, UI는 후순위" |
| **데모·바이럴 각도** | "2주 전 오후 3시 상태로 되돌리기" — 실수 복구 시나리오. 프리미엄 기능을 무료로 제공한다는 메시지. |

---

#### A-3. 편집에도 안 깨지는 앵커 (Stable Anchors)

| 항목 | 내용 |
|------|------|
| **무엇인가** | 댓글, 링크, 커서가 항상 올바른 문자에 붙어 있다. 다른 사람이 앞에 텍스트를 삽입해도 내 커서·댓글이 엉뚱한 위치로 밀리지 않는다. |
| **노션의 약점** | 노션에서 인덱스 기반 커서·댓글은 삽입/삭제가 발생하면 가끔 엉뚱한 위치를 가리킨다. |
| **CRDT가 가능케 하는 이유** | RGA는 모든 문자에 불변 id(`RgaId`)를 부여한다. 커서·댓글은 텍스트 오프셋이 아니라 이 id에 앵커링된다. 링크드 리스트를 직접 탐색하므로 앞뒤 편집에 영향을 받지 않는다([07-collaboration-crdt.md §5-2](./07-collaboration-crdt.md) 참고). |
| **구현 난이도** | ★★★☆☆ — P3 커서 앵커링에 내재. 댓글 앵커 확장은 P5. |
| **적합 Phase** | P3(presence 커서 앵커링에 내재) |
| **데모·바이럴 각도** | "동시 편집 중 커서가 정확히 유지되는" 시연. 기술적 신뢰도 어필. |

---

#### A-4. 문서 브랜치 & 머지 (Git for Docs)

| 항목 | 내용 |
|------|------|
| **무엇인가** | 문서를 브랜치로 복사해 독립적으로 편집한 뒤 원본에 병합한다. Git의 branch/merge와 같은 개념을 문서에 적용한다. |
| **노션의 약점** | 노션에는 이 개념이 없다. 검토용 복사본을 만들고 수동으로 비교·붙여넣기해야 한다. |
| **CRDT가 가능케 하는 이유** | CRDT는 병합이 본질이다. 두 op 로그를 합치는 것이 알고리즘적으로 지원된다. |
| **구현 난이도** | ★★★★☆ — UI/UX 설계가 복잡하고 브랜치 메타데이터 관리 필요. |
| **적합 Phase** | P5+(post-MVP 이후) |
| **데모·바이럴 각도** | 기술 문서 팀, 법률 계약 검토, 교육 자료 버전 관리 등 틈새 시장 어필. |

---

### [B] 협업 UX — 피그마급 멀티플레이어

#### B-5. 관전 모드 (Follow Mode)

| 항목 | 내용 |
|------|------|
| **무엇인가** | 팀원의 시점을 따라가며 같은 위치를 함께 본다. 발표자가 문서를 스크롤하면 관전자 화면도 동기화된다. |
| **노션의 약점** | 노션에는 공식 Follow 모드가 없다. 화상회의를 켜놓고 "지금 어디 보세요?"를 구두로 확인해야 한다. |
| **CRDT가 가능케 하는 이유** | P3 presence 인프라(ws 브로드캐스트 + 커서 앵커)가 기반이다. 뷰포트 위치를 presence 메시지에 포함하면 된다. |
| **구현 난이도** | ★★★☆☆ — presence 메시지 확장 + 클라이언트 스크롤 동기화. |
| **적합 Phase** | P3 확장 또는 P5 |
| **데모·바이럴 각도** | "팀원과 같은 문서를 함께 보며 리뷰" — 화상회의 대체 또는 보완. |

---

#### B-6. 블록 단위 presence 히트맵

| 항목 | 내용 |
|------|------|
| **무엇인가** | 누가 어느 블록을 보고 있는지 시각적으로 표시한다. 블록 옆에 아바타가 보여 "지금 이 단락을 A, B가 보고 있다"를 즉시 파악한다. |
| **노션의 약점** | 노션은 페이지 단위 presence(상단 아바타)만 제공한다. 블록 단위 시각화가 없다. |
| **CRDT가 가능케 하는 이유** | 커서 앵커가 RGA 문자 id이므로 커서 위치 → 블록 인덱스 변환이 자연스럽다. P3 presence 위에 UI 레이어만 추가하면 된다. |
| **구현 난이도** | ★★☆☆☆ — 블록 ID와 커서 앵커 매핑 + 렌더링. |
| **적합 Phase** | P3 확장 또는 P5 |
| **데모·바이럴 각도** | "이 섹션을 5명이 동시에 읽고 있다" — 문서의 살아있는 느낌. |

---

#### B-7. 작성 타임랩스 리플레이 ⭐ wedge

| 항목 | 내용 |
|------|------|
| **무엇인가** | 문서가 작성된 과정을 빠른 영상처럼 재생하고 공유할 수 있다. 커서 이동, 타이핑, 삭제가 시간 순서대로 시각화된다. |
| **노션의 약점** | 노션에는 이 기능이 없다. 버전 히스토리조차 스냅샷 단위이며 "과정"을 볼 수 없다. |
| **CRDT가 가능케 하는 이유** | CrdtOp에는 timestamp가 포함된다. op를 순서대로 재생하면 작성 과정이 재현된다. A-2 타임머신 위에 애니메이션 레이어를 추가하면 된다. |
| **구현 난이도** | ★★★☆☆ — 타임머신 위에 재생 속도 제어 + 공유 링크 생성. |
| **적합 Phase** | P5(post-MVP) — 히스토리 위에 구현, 바이럴 기능 |
| **데모·바이럴 각도** | "우리 팀이 이 기획서를 어떻게 썼는지 공유합니다" — SNS 공유, 교육 콘텐츠, 팀 문화 소재. **Ieum 인지도를 높이는 가장 강력한 바이럴 후크.** |

---

### [C] 더 무거운 것 — 후순위 · 차별성 약함

| 기능 | 설명 | 왜 후순위인가 |
|------|------|--------------|
| 블록 단위 권한 공유 | 특정 블록만 외부에 공개 | 복잡한 ACL 설계 필요. 워크스페이스 권한 모델 안정화 후 검토 |
| 실행되는 코드블록 | 코드 블록 내 코드를 서버에서 실행 | 인프라 복잡도 급증. 차별점이지만 핵심 사용자층이 좁다 |
| 강화 수식 편집기 | LaTeX/KaTeX 인라인 수식 | 틈새 사용자(학술·수학)에만 유효 |
| AI 요약 / AI 편집 보조 | LLM 연동 자동 요약·다음 문장 제안 | 2026년 기준 업계 표준이 돼버려 차별점이 없다. 비용도 문제. |

## 3. 집중 권고 — 3개 wedge로 포지션을 세운다

아래 3개가 Ieum의 포지셔닝을 결정하는 **쐐기(wedge)**다. MVP 완료 후 마케팅·데모·PR의 80%를 이 3개에 집중한다.

| 순위 | Wedge | 핵심 메시지 | 데이터 준비 시점 |
|------|-------|------------|----------------|
| **1** | **로컬 우선 · 오프라인 편집** | "끊겨도 쓴다. 다시 연결하면 알아서 합친다." | P2 코어 완료 후 |
| **2** | **타임머신 버전 히스토리** | "모든 글자의 역사가 남는다. 무료로." | P5 UI 완료 후 |
| **3** | **작성 타임랩스 리플레이** | "문서가 만들어지는 과정을 영상으로 공유하세요." | P5 리플레이 완료 후 |

**왜 이 3개인가**

1. **로컬 우선**: 사용자가 매일 체감하는 UX 차이다. 노션 대비 즉각적인 전환 이유가 된다.
2. **타임머신**: 노션이 유료 기능으로 제공하는 것을 무료로 — 직접적 비용 경쟁력.
3. **타임랩스**: 영상으로 공유 가능 → SNS 바이럴 → Ieum 자체가 마케팅 채널이 된다.

세 개 모두 추가 서버 인프라 없이 **P2 op 로그와 P3 presence 위에** 구현된다. 아키텍처 결정이 이미 이 세 개를 약속한 셈이다.

## 4. 포지셔닝 태그라인 후보

| 번호 | 태그라인 | 강조점 |
|------|---------|--------|
| **A** | "끊겨도 쓰고, 합쳐도 깨지지 않는 문서." | 오프라인·병합 신뢰성 |
| **B** | "모든 글자의 역사를 기억하는 협업 에디터." | 타임머신 히스토리 |
| **C** | "실시간 협업을 아키텍처로 푼다." | 기술 우위 포지셔닝 (개발자·기술팀 타겟) |

> 권고: 초기 랜딩 페이지는 **A** 또는 **B** 중 하나로 시작. **C**는 기술 블로그·PR 채널용.

## 5. 차별점 → Phase 매핑

| 차별점 | 분류 | 적합 Phase | 비고 |
|--------|------|-----------|------|
| 편집에도 안 깨지는 앵커 | [A] | **P3** | presence 커서 앵커링에 내재 |
| 로컬 우선·오프라인 편집 | [A] | **P2-P3 연장 / P5+** | P2 CRDT 코어 위에 op 큐 + IndexedDB 추가; 완전한 local-first 영속화는 post-MVP |
| 타임머신 버전 히스토리 | [A] | **P5** | op 로그는 P2부터 쌓임. "데이터는 공짜, UI는 후순위" |
| 작성 타임랩스 리플레이 | [B] | **P5** | 타임머신 위에 구현. 바이럴 기능. |
| 관전 모드 (Follow) | [B] | **P3 확장 또는 P5** | presence 메시지 뷰포트 확장 |
| 블록 단위 presence 히트맵 | [B] | **P3 확장 또는 P5** | 커서 앵커 → 블록 매핑 추가 |
| 문서 브랜치 & 머지 | [A] | **P5+** | CRDT 병합은 지원, UI/UX 설계 별도 필요 |
| 블록 단위 권한 공유 | [C] | post-P5 | 권한 모델 안정화 후 |
