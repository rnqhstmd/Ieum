## 1. 비전 및 한 줄 정의

> **"누구나 초대하고, 함께 쓰고, 즉시 반영되는 문서 공간 — 그 동작 원리를 직접 만든다."**

Ieum은 Google 계정으로 바로 시작할 수 있는 협업 문서 서비스다. 개인 메모부터 팀 단위 공동 편집까지 하나의 워크스페이스에서 처리한다. 외부 실시간 협업 라이브러리(Yjs 등)에 의존하지 않고, 자체 구현한 RGA CRDT 엔진 위에서 동작한다는 점이 기술적 핵심이다.

---

## 2. 해결하는 문제 — 왜 만드나

### 사용자 관점 문제

| 문제 | 현재 불편함 |
|------|------------|
| 협업 도구 진입 장벽 | 팀 도구를 쓰려면 계정 생성, 조직 설정 등 초기 비용이 크다. Google 계정 하나로 즉시 시작하고 이메일로 초대할 수 있어야 한다. |
| 동시 편집 충돌 | 여러 명이 같은 문서를 편집하면 덮어쓰기나 충돌이 발생한다. 자동으로 수렴하는 편집 경험이 필요하다. |
| 문서 구조 한계 | 단일 평면 목록으로는 지식을 계층적으로 구성하기 어렵다. 페이지 안에 페이지를 두는 트리 구조가 필요하다. |
| 현재 상태 파악 불가 | 누가 지금 이 문서를 보고 있는지, 커서가 어디 있는지 알 수 없다. presence 정보가 필요하다. |

### 학습/공학 관점 문제

시중의 협업 도구는 Yjs, ShareDB 등 검증된 라이브러리를 사용한다. 이 프로젝트는 그 블랙박스를 직접 열어 RGA CRDT를 처음부터 구현하고, TDD로 정확성을 증명하는 것을 목표로 한다. "동작하는 협업 서비스"와 "직접 구현한 협업 엔진"을 동시에 달성한다.

## 3. 타겟 사용자

### 1차 사용자 — 소규모 협업 팀

- 5명 이하의 팀이 공유 문서가 필요하다.
- 별도 계정 생성 없이 Google 계정으로 초대받아 바로 합류한다.
- 회의록, 스펙 문서, 아이디어 노트 등 텍스트 중심 콘텐츠를 공동 편집한다.

### 2차 사용자 — 개인 사용자

- 개인 워크스페이스에서 중첩 페이지 구조로 메모와 문서를 관리한다.
- 외부에 공유할 필요 없이 자신만의 지식 트리를 구성한다.

### 3차 사용자 (학습 목적) — 개발자

- CRDT 알고리즘, 실시간 WebSocket 프로토콜, TDD 방법론을 직접 구현한 레퍼런스로 활용한다.

## 4. 핵심 가치

| 가치 | 설명 |
|------|------|
| **즉시 진입** | Google 계정 하나로 로그인, 워크스페이스 생성, 이메일 초대까지 3분 이내 완료 |
| **충돌 없는 협업** | 네트워크가 일시 단절되거나 동시에 편집해도 모든 클라이언트가 동일한 상태로 수렴 |
| **직관적인 문서 구조** | 페이지를 중첩해 계층을 만들고, 사이드바에서 트리로 탐색 |
| **살아있는 공간** | 아바타, 라이브 커서, "지금 보는 중" 표시로 협업자의 존재를 실시간으로 인식 |
| **투명한 구현** | 협업 엔진 전체가 자체 코드이며, 테스트로 동작을 증명 |

## 5. 경쟁 및 유사 서비스와의 차별점

### 비교 대상

| 서비스 | 공통점 | 차이점 |
|--------|--------|--------|
| **Notion** | 블록 에디터, 중첩 페이지, 협업 | 자체 CRDT 구현 여부 불투명, 학습 목적 코드 공개 없음 |
| **Google Docs** | Google 계정 연동, 실시간 협업 | 문서 구조가 단일 문서 중심, 페이지 트리 없음 |
| **Confluence** | 팀 협업, 문서 계층 | 무겁고 복잡한 설정, 소규모 팀에 과함 |
| **HackMD / CodiMD** | 실시간 공동편집 (Markdown) | Yjs 기반, CRDT 직접 구현 아님 |

### Ieum의 차별화 포인트

1. **자체 RGA CRDT 구현**: Yjs, Automerge 등 외부 CRDT 라이브러리를 사용하지 않는다. RGA 알고리즘을 TypeScript로 직접 구현하고, 단위 테스트로 수렴 정확성을 검증한다.

2. **TDD 전체 적용**: 협업 엔진부터 API, e2e(2-브라우저 동시 편집 시나리오)까지 테스트 우선으로 개발한다. 구현 결과물이 곧 학습 레퍼런스가 된다.

3. **경량 진입**: 조직·빌링·복잡한 권한 체계 없이 Google 로그인 후 즉시 사용한다. MVP에서는 OWNER/MEMBER 두 역할만 존재한다.

4. **공개 학습 자산**: 아키텍처 문서, CRDT 설계 문서, TDD 전략 문서가 코드와 함께 레포에 포함된다.

## 6. 성공 기준

### 정성적 기준

- [ ] Google 로그인 → 개인 워크스페이스 페이지 편집까지 신규 사용자가 5분 이내 도달할 수 있다.
- [ ] 2명이 같은 페이지를 동시에 편집할 때 충돌 없이 내용이 합쳐진다.
- [ ] 라이브 커서와 presence 표시가 체감상 "실시간"으로 느껴진다 (300ms 이내 반영 목표).
- [ ] RGA CRDT 코드와 테스트만 읽어도 알고리즘을 이해할 수 있을 만큼 가독성이 확보된다.

### 정량적 기준 (MVP 완료 시점)

| 지표 | 목표 |
|------|------|
| CRDT 단위 테스트 커버리지 | 주요 경로 100% (INSERT/DELETE/동시편집 충돌 시나리오) |
| Playwright e2e — 2-브라우저 동시편집 시나리오 | 전 케이스 통과 |
| presence 브로드캐스트 지연 | p95 < 300ms (로컬 환경 기준) |
| 초대 → 합류 → 편집 가능까지 스텝 수 | 5스텝 이하 |

## 7. 비범위 (안 만드는 것, MVP 기준)

아래 항목은 의도적으로 MVP에서 제외한다. 기술 부채가 아니라 범위 결정이다.

| 제외 항목 | 이유 |
|-----------|------|
| Viewer 역할 (읽기 전용 멤버) | OWNER/MEMBER 두 역할로 MVP 충분, 추후 추가 |
| 페이지별 개별 공유 / 공개 링크 | 워크스페이스 멤버십 상속으로 단순화 |
| 이미지·파일 첨부, 임베드 블록 | 텍스트 블록 에디터 완성 후 확장 |
| 슬래시(/) 명령 블록 메뉴 | 블록 타입 확장과 함께 post-MVP |
| 댓글·인라인 코멘트 | post-MVP |
| 전문 검색 (Full-text search) | post-MVP |
| 모바일 최적화 UI | 데스크톱 브라우저 우선 |
| 알림(이메일·푸시) | 초대 이메일만 MVP 포함, 그 외 post-MVP |
| 오프라인 모드 (IndexedDB 로컬 캐싱) | 실시간 연결 전제 MVP, 오프라인 지원은 post-MVP |
| 다중 언어(i18n) | 한국어/영어 혼용 허용, 공식 i18n 프레임워크 미적용 |
