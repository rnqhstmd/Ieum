## MVP 범위 요약

### In-Scope (MVP에 포함)

| # | 기능 영역 | 핵심 내용 |
|---|-----------|-----------|
| 1 | 인증 | Google OAuth 로그인/로그아웃, 세션 관리 |
| 2 | 개인 워크스페이스 | 자동 생성, 페이지 CRUD, 중첩 트리 |
| 3 | 페이지 편집 (단일 사용자) | 블록 기반 contenteditable 에디터, 자동저장 |
| 4 | 실시간 공동편집 | RGA CRDT + WebSocket relay, 2인 이상 동시 편집 수렴 |
| 5 | Presence | 라이브 커서, 접속자 아바타, "이 페이지 보는 중" 표시 |
| 6 | 공유 워크스페이스 | 생성, 이메일 초대, OWNER/MEMBER 역할, 멤버 관리 |

### Out-of-Scope (post-MVP)

| 제외 항목 | 비고 |
|-----------|------|
| Viewer 역할 (읽기 전용) | 역할 모델 확장 시 추가 |
| 페이지별 공유 / 공개 링크 | 워크스페이스 단위 권한으로 대체 |
| 슬래시(/) 블록 메뉴, 이미지·파일 블록 | 블록 타입 확장 단계 |
| 댓글·인라인 코멘트 | 협업 심화 단계 |
| 전문 검색 (Full-text search) | 인덱싱 전략 별도 설계 필요 |
| 오프라인 모드 (로컬 캐싱) | 실시간 연결 전제 |
| 알림 (초대 이메일 외) | 초대 이메일만 MVP 포함 |
| 모바일 최적화 | 데스크톱 브라우저 우선 |

## 기능별 요구사항

### 1. 인증 (Authentication)

#### 사용자 스토리

- **US-AUTH-01**: 방문자로서 Google 계정으로 로그인하고 싶다. 왜냐하면 별도 계정을 만들지 않고 바로 서비스를 사용하고 싶기 때문이다.
- **US-AUTH-02**: 로그인한 사용자로서 로그아웃하고 싶다. 왜냐하면 공용 컴퓨터에서 내 세션을 안전하게 종료해야 하기 때문이다.
- **US-AUTH-03**: 로그인하지 않은 사용자로서 보호된 페이지에 접근하려 할 때 로그인 페이지로 이동하고 싶다. 왜냐하면 인증 없이 내 데이터에 접근되는 것을 방지해야 하기 때문이다.

#### 수용 기준 (Acceptance Criteria)

- [ ] Google OAuth 버튼 클릭 시 Google 계정 선택 화면으로 이동한다.
- [ ] 최초 로그인 시 User 레코드가 자동 생성되고 개인 워크스페이스가 함께 생성된다.
- [ ] 재로그인 시 기존 User 레코드를 재사용한다 (googleId 기준).
- [ ] 로그아웃 후 보호된 경로에 접근하면 `/login` 또는 로그인 페이지로 리다이렉트된다.
- [ ] 유효하지 않거나 만료된 세션으로 API 호출 시 401 응답을 반환한다.
- [ ] 로그인 성공 후 이전에 접근하려던 페이지로 돌아간다 (callbackUrl 처리).

### 2. 워크스페이스 (Workspace)

#### 사용자 스토리

- **US-WS-01**: 로그인한 사용자로서 개인 워크스페이스가 자동으로 존재하길 원한다. 왜냐하면 별도 설정 없이 바로 문서 작성을 시작하고 싶기 때문이다.
- **US-WS-02**: 사용자로서 새로운 공유 워크스페이스를 만들고 싶다. 왜냐하면 팀원들과 협업할 공간이 필요하기 때문이다.
- **US-WS-03**: 사용자로서 내가 속한 모든 워크스페이스를 사이드바에서 확인하고 전환하고 싶다. 왜냐하면 개인 작업과 팀 작업을 구분해 관리하고 싶기 때문이다.
- **US-WS-04**: 공유 워크스페이스 OWNER로서 워크스페이스를 삭제하고 싶다. 왜냐하면 더 이상 필요 없는 협업 공간을 정리해야 하기 때문이다.

#### 수용 기준

- [ ] 최초 로그인 시 `type=PERSONAL` 워크스페이스가 1개 자동 생성된다.
- [ ] 개인 워크스페이스는 추가 생성하거나 삭제할 수 없다 (1인 1개 고정).
- [ ] 공유 워크스페이스 생성 시 생성자는 자동으로 OWNER 역할의 Membership이 생성된다.
- [ ] 사이드바에 내가 속한 개인·공유 워크스페이스 목록이 표시된다.
- [ ] 워크스페이스 이름은 1자 이상 100자 이하이다.
- [ ] OWNER만 공유 워크스페이스를 삭제할 수 있다. 삭제 시 하위 페이지와 멤버십이 함께 삭제된다.
- [ ] MEMBER는 공유 워크스페이스를 나갈 수 있다 (Membership 삭제). OWNER가 혼자 남은 경우 나가기를 시도하면 경고를 표시한다.

### 3. 페이지 (Page)

#### 사용자 스토리

- **US-PAGE-01**: 워크스페이스 멤버로서 새 페이지를 만들고 싶다. 왜냐하면 새로운 문서 작업을 시작해야 하기 때문이다.
- **US-PAGE-02**: 사용자로서 페이지 안에 하위 페이지를 만들고 싶다. 왜냐하면 관련 문서를 계층적으로 구성하고 싶기 때문이다.
- **US-PAGE-03**: 사용자로서 사이드바에서 페이지 트리를 탐색하고 싶다. 왜냐하면 중첩된 문서 구조를 한눈에 파악하고 이동하고 싶기 때문이다.
- **US-PAGE-04**: 사용자로서 페이지를 삭제(아카이브)하고 싶다. 왜냐하면 더 이상 필요 없는 문서를 정리하고 싶기 때문이다.
- **US-PAGE-05**: 사용자로서 페이지 제목과 아이콘을 설정하고 싶다. 왜냐하면 트리에서 페이지를 쉽게 구분하고 싶기 때문이다.

#### 수용 기준

- [ ] 워크스페이스 내 최상위 페이지와 특정 페이지의 하위 페이지를 생성할 수 있다.
- [ ] 페이지 트리 깊이 제한 없음 (무한 중첩 허용, MVP 실용 범위 내).
- [ ] 사이드바에서 부모 페이지를 펼치면 하위 페이지 목록이 표시된다.
- [ ] 페이지 삭제는 soft delete(archivedAt 설정) 방식이다. 하위 페이지도 함께 아카이브된다.
- [ ] 아카이브된 페이지는 트리에서 숨겨진다 (MVP에서 복원 UI는 선택적).
- [ ] 페이지 순서(position)를 드래그앤드롭 또는 이동 UI로 변경할 수 있다 (post-MVP로 연기 가능, position 필드는 유지).
- [ ] 페이지 제목 변경은 에디터 상단에서 인라인으로 수행한다.
- [ ] 아이콘은 이모지 문자열로 저장한다 (이모지 피커 UI는 post-MVP, 직접 입력 허용).

### 4. 에디터 (Block Editor)

#### 사용자 스토리

- **US-EDIT-01**: 사용자로서 페이지에서 텍스트를 입력하고 수정하고 싶다. 왜냐하면 문서 내용을 작성해야 하기 때문이다.
- **US-EDIT-02**: 사용자로서 편집 내용이 자동으로 저장되길 원한다. 왜냐하면 저장 버튼을 누르지 않아도 내용을 잃지 않고 싶기 때문이다.
- **US-EDIT-03**: 사용자로서 텍스트 블록, 제목 블록 등 기본 블록 타입을 사용하고 싶다. 왜냐하면 문서 구조를 시각적으로 구분하고 싶기 때문이다.

#### 수용 기준

- [ ] 에디터는 contenteditable 기반 블록 단위 구조로 동작한다.
- [ ] MVP에서 지원하는 블록 타입: 단락(paragraph), 제목 1~3(heading1~3), 글머리 기호 목록(bullet list). 추가 타입은 post-MVP.
- [ ] Enter 키로 새 블록을 생성한다. Backspace로 빈 블록을 삭제한다.
- [ ] 편집 내용은 debounce(예: 500ms) 후 자동저장되거나, 실시간 협업 시 CRDT op로 즉시 전송된다.
- [ ] 단일 사용자 모드(협업 없음)에서도 에디터는 동일하게 동작한다.
- [ ] 에디터 내용은 RGA CRDT 상태로 관리하며, 렌더링은 CRDT 상태에서 파생한다.

### 5. 실시간 공동편집 (Real-time Collaboration — CRDT)

#### 사용자 스토리

- **US-CRDT-01**: 같은 페이지를 편집 중인 팀원으로서 내 편집이 상대방 화면에 즉시 반영되길 원한다. 왜냐하면 충돌이나 덮어쓰기 없이 함께 작업하고 싶기 때문이다.
- **US-CRDT-02**: 사용자로서 네트워크가 일시 끊겼다 재연결됐을 때 편집 내용이 유실되지 않길 원한다. 왜냐하면 작업한 내용을 잃고 싶지 않기 때문이다.
- **US-CRDT-03**: 개발자로서 CRDT op 로그를 통해 편집 이력을 추적하고 싶다. 왜냐하면 상태 재현과 디버깅이 필요하기 때문이다.

#### 수용 기준

- [ ] 2명 이상이 같은 페이지를 동시에 편집할 때 모든 클라이언트가 동일한 최종 텍스트 상태로 수렴한다.
- [ ] 편집 op는 WebSocket을 통해 relay 서버로 전송되고, 서버가 같은 페이지의 다른 클라이언트에게 broadcast한다.
- [ ] 모든 op는 `CrdtOp` 테이블에 append-only로 저장된다.
- [ ] 신규 접속 클라이언트는 서버에서 현재 전체 상태(snapshot 또는 op 재생)를 받아 초기화된다.
- [ ] RGA INSERT op는 `{siteId, seq, afterId, value}` 구조를 갖는다.
- [ ] RGA DELETE op는 `{targetId}` 구조로 해당 문자를 tombstone 처리한다.
- [ ] 동일 위치 동시 삽입은 `siteId` 기준 결정론적 순서로 해소된다 (모든 클라이언트 동일 결과).
- [ ] CRDT 단위 테스트: INSERT/DELETE/동시 삽입 충돌/tombstone 재삽입 시도 등 주요 경로를 커버한다.
- [ ] Playwright e2e: 브라우저 2개에서 동시 편집 후 양쪽 텍스트가 동일함을 확인한다.

### 6. Presence

#### 사용자 스토리

- **US-PRES-01**: 공동 편집 중인 사용자로서 현재 같은 페이지를 보고 있는 사람들의 아바타를 확인하고 싶다. 왜냐하면 누가 함께 작업 중인지 알고 싶기 때문이다.
- **US-PRES-02**: 사용자로서 협업자의 커서 위치가 에디터에 표시되길 원한다. 왜냐하면 상대방이 어디를 편집하고 있는지 파악하고 싶기 때문이다.
- **US-PRES-03**: 사용자로서 페이지를 열면 "이 페이지를 보는 중" 표시가 나타나고, 페이지를 닫으면 사라지길 원한다. 왜냐하면 현재 활성 사용자를 실시간으로 파악하고 싶기 때문이다.

#### 수용 기준

- [ ] 페이지 접속 시 relay 서버에 presence 참여를 알린다. 탭 종료 또는 페이지 이동 시 이탈을 알린다.
- [ ] 현재 페이지의 접속자 목록(이름, 아바타 이미지)이 에디터 상단에 표시된다.
- [ ] 협업자의 커서 위치가 에디터 내에 컬러 표시(이름 레이블 포함)된다.
- [ ] 커서 위치는 RGA 노드 ID(커서 앵커)로 표현되어, op 적용 후에도 올바른 위치를 유지한다.
- [ ] presence 업데이트는 커서 이동 이벤트 시 debounce(50ms) 후 전송한다.
- [ ] 사용자가 페이지를 떠나면 해당 사용자의 커서가 즉시 사라진다 (relay 서버의 disconnect 처리).
- [ ] presence 정보는 서버에 영속 저장하지 않는다 (메모리 내 상태, 재연결 시 재구독).

### 7. 초대 및 역할 관리 (Invitation & Permissions)

#### 사용자 스토리

- **US-INV-01**: 공유 워크스페이스 OWNER로서 이메일 주소로 팀원을 초대하고 싶다. 왜냐하면 협업할 사람을 워크스페이스에 추가해야 하기 때문이다.
- **US-INV-02**: 초대받은 사용자로서 이메일의 초대 링크를 클릭해 워크스페이스에 합류하고 싶다. 왜냐하면 초대를 수락하고 바로 작업을 시작하고 싶기 때문이다.
- **US-INV-03**: OWNER로서 MEMBER의 역할을 변경하거나 멤버를 내보내고 싶다. 왜냐하면 워크스페이스 멤버십을 관리해야 하기 때문이다.
- **US-INV-04**: OWNER로서 보류 중인 초대를 취소하고 싶다. 왜냐하면 잘못 초대했거나 더 이상 초대가 필요 없는 경우 정리해야 하기 때문이다.

#### 수용 기준

- [ ] OWNER는 이메일 주소를 입력해 초대를 생성할 수 있다. 초대 이메일이 해당 주소로 발송된다.
- [ ] 초대 링크는 고유 token을 포함하며 7일 후 만료된다 (expiresAt).
- [ ] 초대 링크 클릭 시 로그인 상태면 즉시 Membership이 생성되고, 미로그인이면 로그인 후 처리된다.
- [ ] 이미 멤버인 이메일로 초대 시 에러 메시지를 반환한다.
- [ ] 초대 상태는 `PENDING → ACCEPTED | REVOKED | EXPIRED` 흐름을 따른다.
- [ ] OWNER는 PENDING 상태의 초대를 REVOKED로 변경(취소)할 수 있다.
- [ ] OWNER는 MEMBER를 워크스페이스에서 내보낼 수 있다 (Membership 삭제).
- [ ] OWNER는 MEMBER에게 OWNER 역할을 부여할 수 있다.
- [ ] 워크스페이스에는 OWNER가 최소 1명 이상 존재해야 한다. 마지막 OWNER는 역할을 변경하거나 나갈 수 없다.
- [ ] MEMBER는 워크스페이스 내 모든 페이지를 편집할 수 있다.
- [ ] 워크스페이스에 속하지 않은 사용자는 해당 워크스페이스의 페이지에 접근할 수 없다 (403 반환).

## 비기능 요구사항

### 성능

| 항목 | 목표 | 측정 방법 |
|------|------|-----------|
| 페이지 초기 로드 (에디터 표시까지) | < 2초 (로컬 환경) | Playwright 타이밍 |
| CRDT op 브로드캐스트 지연 | p95 < 300ms (로컬 환경) | WebSocket 타임스탬프 측정 |
| presence 업데이트 지연 | p95 < 300ms (로컬 환경) | WebSocket 타임스탬프 측정 |
| 동시 접속자 (MVP 목표) | 워크스페이스당 10명 | 부하 테스트 (post-MVP) |

### 동시성

- 같은 페이지에 대한 동시 편집 op는 CRDT 알고리즘으로 수렴을 보장한다. DB 레이어에서 op는 append-only이므로 write 충돌이 발생하지 않는다.
- Invitation token 중복 사용 방지: token 기반 조회 후 상태 전환을 단일 트랜잭션 내에서 처리한다.

### 보안

- 모든 API Route Handler는 세션 유효성을 검증한다. 인증되지 않은 요청은 401을 반환한다.
- 워크스페이스 자원(페이지, 멤버, 초대)에 대한 접근은 요청자의 Membership을 확인한다 (403 처리).
- Invitation token은 충분한 엔트로피를 가진 무작위 값으로 생성한다 (예: `crypto.randomUUID()` 기반).
- Google OAuth에서 수신한 이메일과 googleId 외의 민감 정보는 저장하지 않는다.
- WebSocket 연결 시 세션 쿠키 또는 토큰으로 사용자를 인증한다. 미인증 WebSocket 연결은 거부한다.

### 접근성

- 에디터 및 주요 인터랙션은 키보드 탐색을 지원한다.
- 이미지(아바타 등)에는 alt 텍스트를 제공한다.
- 색상 대비는 WCAG AA 기준을 충족한다 (MVP 범위 내 구현 노력 기울임, 전체 감사는 post-MVP).
